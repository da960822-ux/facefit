## 프로젝트 개요

FaceFit은 취업 준비자가 이력서와 지원 맥락을 바탕으로 AI 면접관과 음성 모의면접을 진행하고, 답변 내용·발화·시선·자세에 대한 근거 기반 피드백을 확인하는 데스크톱 웹 서비스다. 단발성 점수 제공이 아니라 설정→실전 연습→분석→리포트→재연습의 개선 루프를 설계했다.

## 문제 정의와 핵심 사용자

면접 준비자는 예상 답안을 작성해도 변형 질문에 자신의 경험을 자연스럽게 설명하기 어렵다. 셀프 녹화는 관찰 기준이 없고, 단순 점수는 무엇을 고쳐야 하는지 알려주지 않는다. 첫 면접 준비자, 직무 전환자, 반복 개선자를 행동과 상황 기준으로 나눠 “실제 흐름에 익숙해지기”, “경험을 직무 언어로 설명하기”, “개선 여부 확인하기”를 핵심 JTBD로 정의했다.

## 핵심 의사결정

### 1. 점수가 아니라 근거와 다음 행동을 제품 중심에 놓았다

4축 분석을 내용·발화·시선·자세로 제한하고, 각 결과가 답변 문장·발화 구간·관찰 구간과 연결되도록 요구했다. 감정·성격·합격 가능성은 평가에서 제외했다. 이는 사용자가 결과를 행동으로 바꾸고, 서비스가 진단 도구로 오해되는 위험을 줄이기 위한 결정이다.

### 2. 답변 완료는 사용자가 통제하고 침묵 감지는 보조로 뒀다

5초 침묵 자동 완료만 사용하면 생각하는 멈춤이나 소음을 답변 종료로 오인할 수 있다. 버튼과 Space를 같은 완료 동작으로 통합하고, 연타·입력 포커스·모달·저장 중에는 중복 실행을 막도록 정의했다.

### 3. 실시간 아바타의 품질 실패를 제품 실패로 만들지 않았다

OpenAvatarChat+MuseTalk을 목표 면접관으로 두되 V1은 반이중 대화를 우선했다. 영상 실패 시 음성+정적 화면, 음성 실패 시 질문 텍스트로 전환해 핵심 연습을 유지하도록 했다.

### 4. 사실·가정·협의를 분리했다

프론트엔드를 React+Vite로 전환하면서 기존 화면·행동·URL을 React Router SPA로 보존했다. `/pricing`은 존재하지만 결제는 범위 밖이라는 점도 명확히 기록했다. API 명세 전에는 엔드포인트와 JSON을 만들지 않고 필요한 상태·입출력·실패 경계만 정의했다.

## 협업 방식과 개발자 조율

- 프론트는 입력·권한·상태·복구 UI, AI·백엔드는 세션·모델·저장·분석 상태를 소유하도록 경계를 나눴다.
- 실제 API 전에 연동 ID로 필요한 입력·출력·실패·타임아웃·재시도 질문을 준비했다.
- UX에는 화면별 빈·오류·로딩·포커스 상태를 전달하고, QA에는 Given-When-Then 수용 기준과 34개 테스트 시나리오를 연결했다.
- 결정이 바뀌면 기존 기록을 삭제하지 않고 재검토 조건과 새 결정 ID를 남기는 방식으로 변경 이력을 관리한다.

## 범위·리스크 관리

결제·이용권·관리자·모바일 앱·감정 진단을 V1에서 제외했다. V1은 5문항 반이중 면접과 설명 가능한 4축 리포트에 집중한다. 주요 리스크는 GPU 지연, VAD 오탐, ASR/CV 편차, 민감 데이터, 목업 오해이며 각 리스크를 QA·폴백·정책 결정에 연결했다.
