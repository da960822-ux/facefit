# Face Fit MVP 로드맵

> 기준일: 2026-07-20  
> 예상 기간: 8~10주  
> 기준: 데모 화면을 실제 데이터·실시간 아바타·사후 분석과 연결하는 순서

## 1. 단계별 계획

| 단계 | 기간 | 핵심 결과 | 종료 조건 |
| --- | --- | --- | --- |
| 0. 기준선·환경 | 1주 | 데이터 계약, 동의 정책, GPU·RTC PoC 환경 | 한 턴 A/V 왕복과 추적 ID 확인 |
| 1. 반이중 아바타 면접 | 2주 | OpenAvatarChat + MuseTalk, 1명 면접관, 5문항 | 15분 세션 10회 연속 완료 |
| 2. 제품 흐름 연결 | 2주 | 설정·장비·라이브·분석 상태·리포트 API 연결 | P0 화면에서 고정 데이터 제거 |
| 3. 4축 분석·설명 리포트 | 2주 | CV·발화·내용 분석, 근거·신뢰도·루브릭 | 샘플 100%에 근거 연결 |
| 4. 안정화·사용자 검증 | 1~2주 | 지연 최적화, 폴백, 삭제·장애 QA, 사용자 테스트 | PRD 출시 수용 기준 충족 |
| 5. P1 실험 | 이후 | 수동/전이중 인터럽트, 역질문, 재연습 최적화 | 가드레일 지표 충족 시 점진 공개 |

## 2. 단계 0: 기준선과 환경

### 작업

- `session_id`, `turn_id`, `rubric_version` 데이터 계약 확정
- RTC 토큰 발급과 TURN 연결 경로 구성
- OpenAvatarChat 공식 MuseTalk 설정으로 GPU 단일 세션 실행
- 기술·HR·경영진 아바타 원본 권리와 제작 기준 확정
- 미디어 동의, 보존기간, 삭제 범위 확정
- 프론트·실시간 서비스·분석 워커의 오류 코드 정의

### 산출물

- 한 턴 성능 기준선
- 아바타 전처리 캐시
- 세션·턴 API 스키마
- 개인정보 데이터 맵

## 3. 단계 1: 반이중 아바타 면접

### 단계 1 P0 작업

- OpenAvatarChat RTC, Silero VAD, 실시간 ASR, LLM, TTS, MuseTalk 핸들러 연결
- 면접관 유형 1종, 일반 강도, 5문항 고정 흐름
- 5초 침묵과 버튼·Space 종료
- 질문 자막, 다시 듣기, 면접 종료
- 턴별 지연과 GPU 메트릭 수집
- MuseTalk 실패 시 정적 이미지 + TTS 폴백

### 확장 작업

- 면접관 3종과 강도 2종
- 답변 기반 꼬리질문 0~1개
- 수동 인터럽트

## 4. 단계 2: 제품 흐름 연결

### 단계 2 P0 작업

- 로그인과 사용자 동의
- 기업·직무·이력서·공고·면접관·강도 저장
- 카메라·마이크·스피커 권한과 장치 선택
- 세션 생성·RTC 연결·진행 상태 저장
- 턴별 녹화 업로드와 재시도
- 분석 중·지연·실패·완료 상태
- 마이페이지 최근 기록과 리포트 진입

### 제외

- 음성 복제
- 결제와 기관 구독
- 다중 면접관
- 모바일 라이브 면접 최적화

## 5. 단계 3: 4축 분석과 설명 리포트

### 단계 3 P0 작업

- 영상 품질 검증과 얼굴 미검출 구간 처리
- MediaPipe 시선·고개·어깨 특징 추출
- Faster-Whisper 전사·필러·속도·침묵 분석
- 질문 의도·답변 구조·구체성·근거 분석
- 축별 신뢰도와 제외 규칙
- 4축 점수, 측정 근거, 개선 행동
- 리포트 스키마와 루브릭 버전 저장

### P1 작업

- 질문별 개선 답변
- 기업·직무 기반 역질문 추천
- 같은 조건 재면접
- 회차별 성장 비교

## 6. 단계 4: 안정화와 검증

### 성능

- 15분 세션 반복·부하 테스트
- P50/P95 턴 지연 최적화
- A/V drift와 드롭 프레임 측정
- GPU OOM, 프로세스 재시작, 캐시 미스 대응

### 품질

- 면접관 유형·강도 프롬프트 회귀 테스트
- 저조도, 얼굴 가림, 무음, 강한 소음, 빠른 말하기 테스트
- 전문가 이중 평가와 루브릭 일치도 확인
- 리포트 이해도 사용자 테스트

### 보안·개인정보

- 동의 없는 저장 차단
- 원본·파생 데이터 삭제
- 서명 URL 만료와 권한 분리
- 로그 민감정보 검사
- OSS·모델·데이터 라이선스 목록 확정

## 7. 우선순위 백로그

### P0 - MVP 출시 차단

- 세션 생성과 RTC 연결
- 반이중 5문항 MuseTalk 면접
- 수동 답변 완료와 5초 침묵
- 턴별 미디어 업로드
- 4축 근거·신뢰도·개선 행동
- 분석 비동기 상태와 재시도
- 미디어 동의·삭제·보존 정책
- 정적 이미지 + 음성 폴백

### P1 - MVP 직후

- 꼬리질문 0~1개
- 기술·HR·경영진 + 일반·압박 조합
- 수동 인터럽트
- 역질문 추천
- 같은 조건 재면접과 성장 분석
- 질문별 개선 답변 TTS

### P2 - 검증 후

- 전이중 인터럽트
- 사용자 음성 프로필·음성 복제
- 비개발 직무 확장
- 대학·기관 구독과 관리자 기능
- 발표·스피치 훈련 모드

| 리스크 | 가능성/영향 | 조기 신호 | 완화 |

| 영상·음성 개인정보 우려 | 중/상 | 동의 이탈·삭제 문의 증가 | 최소 수집, 짧은 보존, 명시적 삭제, 로컬 전처리 확대 |
| 음성 복제 오용 | 중/상 | 동의·권리 범위 불명확 | P2 격리, 워터마크·감사·철회·파생 삭제 |
| OSS·모델 라이선스 혼선 | 중/상 | 부속 모델 출처 누락 | 모델 BOM, 테스트 데이터 제외, 법무 검토 |

## 9. 주간 완료 정의

- 코드 또는 설정만 머지한 상태는 완료가 아니다.
- 해당 사용자 흐름의 성공·실패·재시도·폴백을 시연할 수 있어야 한다.
- `session_id`와 `turn_id`로 로그를 추적할 수 있어야 한다.
- 보안·개인정보 영향과 데이터 보존을 확인해야 한다.
- 수용 기준을 자동 또는 반복 가능한 수동 테스트로 남겨야 한다.
